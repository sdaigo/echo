# ワークフロールール

Claude Codeとの協働における行動原則。

## 基本フロー

1. **ドキュメント作成**: 永続ドキュメント（`docs/`）で「何を作るか」を定義
2. **作業計画**: ステアリングファイル（`.steering/`）で「今回何をするか」を計画
3. **実装**: tasklist.mdに従って実装し、進捗を随時更新
4. **検証**: テストと動作確認
5. **更新**: 必要に応じてドキュメント更新

## Do's

- 常にアクセシビリティを考慮すること
- 変更の際は必ず理由を説明すること
- 複雑な提案の前にPlan Modeで承認を得ること
- 既存ドキュメントがある場合はその内容を最優先する
- 成果物を作成したらユーザーに確認を依頼し、明示的に承認されるまで次に進まない
- フィードバックを反映したら再度承認を求める

## Don'ts

- 独自の用語を勝手に作らないこと
- コンテキストが不明な状態で、推測のみで仕様を決定しないこと
- セキュリティ、アクセシビリティを無視したプロトタイプを作成しないこと

## 成果物の自己レビュー

- 出力したコードやドキュメントに対し、「不備やリスクがないか」を3点指摘してから提出すること
- プロトタイプの場合は、テスト実行やブラウザ確認を必須とする

## コンテキストの継続的改善

- 作業中に用語の誤解やプロセスの不備が見つかった場合、直ちにドキュメントを更新し、負債を残さないこと

## スキル共通ルール

- 新規作成時はスキルのテンプレートとガイドを参照する
- 更新時は既存の構造と内容を維持しながら更新する
